### 7.3 自定义节点说明

#### `gt_tf_publisher.py` (Python 3)
- **功能**: 订阅 Gazebo 真值 topic，将其发布为 TF 坐标变换
- **输入**: `/gazebo/ground_truth/state` (`nav_msgs/Odometry`)
- **输出**: TF 变换 `world → ground_truth`
- **用途**: 使 RViz 能在统一坐标系下显示真值位姿

#### `me5413_eval.py` (Python 3)
- **功能**: 在线计算估计位姿与真值位姿之间的平面欧氏距离误差
- **输入**: `/gazebo/ground_truth/state` + 可配置估计话题（默认 `/odom`）
- **输出**: `/me5413/mapping_error` (`std_msgs/Float32`)
- **注意**: 当前比较 `/odom` 时反映里程计漂移，非完整 SLAM 精度

#### `goal_publisher_node.cpp` (C++，已标注 deprecated for 2526)
- **功能**: 从 RViz 面板接收目标名称，转换到 map 坐标系并发布；同步计算机器人到目标的位置/朝向误差
- **输入**: `/rviz_panel/goal_name`、`/gazebo/ground_truth/state`、`/gazebo/ground_truth/box_markers`
- **输出**: `/move_base_simple/goal`、绝对/相对位置误差和朝向误差（4个 Float32 话题）

#### `object_spawner_gz_plugin.cpp` (C++ Gazebo WorldPlugin)
- **功能**: 在 Gazebo 内动态管理场景对象（数字箱子、交通锥、动态障碍）
- **关键机制**:
  - 随机生成4种标签（从1-9中选4）的数字箱，每种数量也随机（保证唯一解）
  - 在上层走廊随机生成交通锥（决定机器人入口选择）
  - 生成动态障碍红色圆柱体（沿 Y 轴来回运动，速度 2.0 m/s）
  - 监听 `/cmd_unblock` 以控制坡道入口锥的打开（10秒窗口）
- **发布**: `/gazebo/ground_truth/box_markers` (`visualization_msgs/MarkerArray`)

#### `rviz_panel` (C++ RViz Plugin, Qt)
- **功能**: 在 RViz 中提供 GUI 控制面板
- **发布**: `/rviz_panel/goal_name` (目标选择)、`/rviz_panel/respawn_objects` (重新生成对象)

## 9. SLAM 方案调研与对比 (For Report)

*更新时间: 2026-04-02*

### 8.1 各方案优劣对比

| 方案 | 类型 | 回环检测 | 坡道表现 | 当前状态 |
|------|------|---------|---------|---------|
| GMapping | 2D RBPF SLAM | 无 | 扫到坡面当墙，严重遮挡 | 已有初步地图，作为 baseline |
| FAST-LIO2 | 3D 紧耦合 LiDAR-IMU | **无** | 可建 3D 地图覆盖坡道 | 同学已测试，第二层畸变严重，**调研失误，放弃** |
| LIO-SAM | 3D 松耦合 LiDAR-IMU + 因子图 | **有** | 可建 3D 地图 + 回环纠正 | 同学切换中，待评估 |

### 8.2 坡道问题专项分析（关键工程难点）

**问题描述**: 坡道在 2D LiDAR 扫描投影到平面后，斜面会产生"虚假障碍点"，move_base 会将其视为墙，导致机器人无法规划通过坡道的路径。

**候选解决方案（由简到复）**:

1. **开环过坡（当前首选方案）**: 在状态机中检测到"准备上坡"状态后，直接发 `/cmd_vel` 定速前进（绕开 move_base），上坡完成后重新激活 AMCL 定位和 move_base。
   - 优点：无需修改导航栈，实现简单
   - 风险：里程计漂移，上坡后位姿偏差大，需要 AMCL 重新收敛
   - 同学评价：「先试一下开环走上去之后再重新定位」

2. **代价地图清除（costmap clearing）**: 上坡区域设为"可行区域"，禁用该区段的障碍层。
   - 需要对 move_base 的 costmap 参数做分区配置

3. **3D 导航（LIO-SAM + 3D costmap）**: 代价最高，复杂度显著上升，当前阶段**不考虑**。
   - 同学评价：「感觉复杂度上升很多 确实没必要因为一段上坡搞3D定位了」

**当前决策**: 优先实现方案1（开环 + 重定位），作为最小可行方案。

## 10. 箱子计数模块实现方案 (Box Counting Module)

*更新时间: 2026-04-09 | 状态: **重大升级——DBSCAN聚类深度估计，手动驾驶+hdl定位模式***

### 10.1 模块定位

- **不依赖地图**：纯感知模块，仅需 Gazebo 仿真运行（`world.launch`）
- **不依赖自主导航**：由 `patrol_node.py` 驱动自动巡逻采集，或手动遥控均可
- **输入**：`/front/image_raw`（相机）+ `/mid/points`（Velodyne 3D点云）+ `/gazebo/ground_truth/state`
- **输出**：各编号箱子数量 `/me5413/box_count`，RViz 标记 `/me5413/box_markers`，调试图 `/me5413/debug_image`

### 10.2 最终技术栈（2026-04-09 更新）

| 组件 | 技术选型 | 理由 |
|------|---------|------|
| 数字识别 | **YOLOv8n**（自训练） | mAP50=98.9%，推理 3.7ms/帧，远优于 EasyOCR |
| 深度/位置估计 | **DBSCAN 3D 聚类 + 投影匹配** | 识别箱子几何cluster中心，远优于单点角度查表 |
| 定位来源 | **hdl_localization TF**（`velodyne→map`） | 手动驾驶时 odom/TF 链正常，无需 Gazebo 真值 |
| 去重逻辑 | 世界坐标欧氏距离（同数字3m，不同数字0.4m） | 防止同箱多角度重复计数 |
| 多观测确认 | ≥2 次、来自相距 ≥0.5m 的不同位置 | 过滤单帧误检 |
| 驾驶方式 | 手动键盘遥控 | 替代 patrol_node 传送模式，更贴近实际任务场景 |

### 10.3 核心算法流程（DBSCAN聚类版）

```
/front/image_raw ──→ YOLOv8n ──→ [(数字, bbox, conf)]
                                         │
/mid/points ──→ 高度过滤(0.05~1.2m) ──→ DBSCAN聚类(eps=0.25m)
             ──→ 尺寸过滤(0.2~2.0m)  ──→ 候选cluster列表
                                         │
             对每个bbox：把cluster投影到图像平面(u=FX*(-vy)/vx+CX)
             找投影点在bbox内且最近bbox中心的cluster
                                         │
             cluster 3D中心 ──→ TF(velodyne→map) ──→ (wx, wy)
                                         │
                    多观测候选池（≥2位置确认 → 加入counted列表）
                                         │
                              ┌──────────┴──────────┐
                         新确认箱子               候选/重复
                           计数+1                  忽略
                              │
                         counter = {'1':N1, '3':N3, ...}
                              │
                         找 min(counter) → 目标房间
```

### 10.4 关键参数

```python
IMAGE_WIDTH    = 640
FX = FY        = 554.254       # 相机内参
CX, CY         = 320, 256
min_conf       = 0.9           # YOLO 置信度阈值（较高，减少误检）
dedup_same_digit = 3.0         # 同数字去重距离（米）
dedup_any_digit  = 0.4         # 不同数字去重距离（米）
min_obs        = 2             # 最少独立观测次数
min_obs_dist   = 0.5           # 独立观测最小间距（米）
# DBSCAN参数
eps            = 0.25          # 聚类邻域半径（米）
min_samples    = 3             # 最少邻居数
cluster_range  = 0.2~2.0m      # 箱子cluster水平尺寸范围
```

### 10.5 YOLOv8 自训练流程（关键工程难点）

**问题**: EasyOCR 在远距离/角度下漏检严重，且无法区分箱子整体与数字区域。

**解决方案**: Gazebo 地面真值 3D 投影自动标注

1. `collect_training_data.py`：
   - 订阅 `/gazebo/model_states` 获取箱子真实 3D 坐标（模型名 `number{d}_{i}` → 数字 d）
   - 将箱子 8 个角点投影到相机图像（pinhole 模型），生成覆盖整个箱子的 YOLO bbox
   - 同房间过滤（隔墙 x≈-10 两侧分开标注，避免墙后不可见目标被标注）
   - 采集结束自动生成 preview 可视化验证标注质量
2. `patrol_node.py` 驱动多角度采集（每个检查点 10 步×9° = 90°，全传送）
3. 多次 respawn 覆盖全部 9 种数字（每次随机出 4 种）
4. 共采集 **1191 张图 / 3479 个标注**，9 种数字全覆盖
5. 训练：`yolo train epochs=100 batch=16`，**mAP50=98.9%，mAP50-95=95.0%**

## 11. 难点与解决记录 (For Report — 重要，写报告时参考)
- **难点**: Gazebo 真值是 ROS topic，不是 TF 坐标系，不能直接在 RViz 坐标框架中使用。
- **解决思路**: 编写 `gt_tf_publisher.py`，将 `/gazebo/ground_truth/state` 中的 `nav_msgs/Odometry` 位姿转换为 `world -> ground_truth` TF。
- **难点**: 在线误差监控和完整 SLAM 评估不是一回事，容易混淆。
- **解决思路**: 先用 `me5413_eval.py` 比较估计位姿与真值，快速观察系统漂移；后续再用 EVO 做离线轨迹评估，形成更严谨结论。
- **难点**: 项目目录中的 `.venv` 会干扰 ROS Python 工具链，导致 `rospkg` 等模块解析失败。
- **解决思路**: 删除项目本地 `.venv`，统一使用 ROS 系统环境与 catkin 工作区环境，避免 `roslaunch` 被错误 Python 环境接管。
- **难点**: 课程仓库没有直接给出官方 2D ground-truth map，但展示与评估又需要一个静态参考视角。
- **解决思路**: 先接受”pseudo ground truth”策略，在尽量静态、低干扰的环境中生成一张高质量 `reference_map`，用于双视角展示和初步对比；同时保留 EVO 与位姿误差作为更正式的量化依据。
- **难点**: `set_model_state` 传送机器人后，`ekf_localization` 不感知传送，导致 odom/TF 链给出错误的世界坐标，箱子被重复计数（同一箱子计数18次）。
- **解决思路**: 新增 `use_ground_truth` 模式，直接订阅 `/gazebo/ground_truth/state` 获取真实位姿，完全绕过 TF 链，彻底解决传送后坐标错误问题。
- **难点**: 2D SICK TIM551 激光在箱子间隙处返回 inf，导致大量 NO_POS（无法定位），目标物体无法被计入。
- **解决思路**: 换用 Velodyne 16线3D点云（`/mid/points`），建立水平角极坐标查找表（0.5°/bin），±10°窗口搜索最近点，覆盖全方向且无间隙问题。
- **难点**: EasyOCR 远距离/斜角漏检严重，conf<0.75 的箱子全部丢失，训练数据自标注质量差。
- **解决思路**: 改用 Gazebo 地面真值 3D 投影自动标注（`collect_training_data.py`），将箱子8角投影到图像生成精确bbox，标注准确率100%，不受距离影响；自训练 YOLOv8n 达到 mAP50=98.9%。
- **难点**: 传送后 cmd_vel 旋转不生效（物理引擎状态异常），导致10步扫描全在同一角度，训练数据视角单一。
- **解决思路**: `patrol_node.py` 的旋转步骤也改为 `set_model_state` 直接传送到每个目标角度，保证每步角度精确，完全绕开 cmd_vel。
- **难点**（2026-04-09）：点云深度估计不准——单点角度查表法选到墙面背景点，导致检测位置严重偏移，去重失效（同一箱子计数16次）。同时 Velodyne 角度符号约定（+y=左）与图像坐标（+x=右）相反，若不取反则查找方向完全错误。
- **解决思路**: 改用 DBSCAN 聚类（eps=0.25m）对高度过滤后的点云做水平面聚类，识别出几何上属于同一物体的点群，取cluster中心作为箱子3D坐标。再将cluster中心投影到图像（`u=FX*(-vy)/vx+CX`，注意 y 取反），找与 YOLO bbox 中心最近的cluster完成匹配。cluster中心稳定，不受单帧背景点干扰。
- **难点**（2026-04-09）：代码中遗留废弃字段 `_cloud_dist`（始终为 None），`_run_detection` 检查该字段导致检测逻辑完全不执行，表现为节点无任何输出，极难发现。
- **解决思路**: 改为检查 `_last_cloud_time is None`，确认点云已接收即可进入检测逻辑。
- **难点**（2026-04-09）：相邻的同数字箱子（如两个"8"，间距1.5~2m）中间那个始终无法被计入。
- **解决思路**: 两个根因——①多个 bbox 串行抢同一个 cluster，改为全局一对一贪心排他分配；②`dedup_same_digit=3.0m` 太大，DBSCAN 精度已达 ≤0.5m，降至 1.2m 后两个相邻同数字箱均可正常计入。
- **难点**（2026-04-09）：debug image 站着不动时完全冻结，只有机器人移动 ≥0.3m 才刷新一次，导致调试极其不便。
- **解决思路**: 新增独立 5Hz 显示定时器（`_display_cb`），只做 YOLO 画框+发图，不碰候选池/计数状态，与计数定时器完全解耦。
